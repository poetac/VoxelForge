# Architecture Decision Records

This folder captures the load-bearing architectural decisions behind
`Voxelforge`. Each ADR is a small document (context / decision /
alternatives / consequences / status) so a future reader can understand the
*why* without excavating sprint notes.

## Conventions

- ADRs are numbered sequentially from 001. **Numbers are stable** —
  removed ADRs leave a gap; renumbering would break inbound references
  in commit messages and the project docs.
- Status values: **Proposed**, **Accepted**, **Open**, **Deprecated**,
  **Resolved**, **Superseded** (by ADR-XXX). A parenthetical qualifier
  like `Accepted (debt)` or `Accepted (with exit path flagged)` is
  allowed when the ADR captures a decision that is owned, documented,
  and stable but carries a tracked architectural cost.
- ADRs are immutable once Accepted. To change a decision, write a new
  ADR that supersedes the old one. **Refreshing observable context**
  (test counts, line numbers, file paths, sprint references) inside an
  ADR is NOT a decision change and is permitted without a superseding
  ADR — drift between the ADR and the live codebase wastes a future
  reader's time.
- An ADR may be **removed entirely** (rather than left as a Superseded
  stub) when its content is dead — i.e. the decision is no longer in
  force AND no future reader is likely to need the historical context.
  Removed ADRs leave a row in the "Removed ADRs" table above with a
  one-line justification. Git history is the canonical record of the
  former content.
- Keep each ADR under ~80 lines where practical. If it wants to be
  longer, split it.
- Mention specific file paths and test names — ADRs are referenced from
  the project docs and should be concrete.
